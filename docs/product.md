# 产品说明

## 1. 产品定位

萤石智宠是一个面向家庭宠物看护场景的智能监控系统，围绕“设备接入、实时监看、宠物档案、安全区域检测、异常告警、AI 辅助分析”构建第一阶段可演示闭环。

第一版 Demo 的目标不是覆盖完整商用能力，而是验证以下核心价值：

- 用户可以绑定并查看自己授权的萤石设备
- 用户可以基于设备建立宠物检测配置
- 系统可以基于截图和 AI 识别结果生成检测记录与异常告警
- 用户可以在前端完成查看、筛选、追踪和基础运维动作

## 2. 角色与权限

当前系统定义三类角色：

| 角色 | 说明 | 权限 |
|------|------|------|
| `ADMIN` | 管理员 | 当前版本下拥有读写权限 |
| `OPERATOR` | 业务操作人员 | 当前版本下拥有读写权限 |
| `VIEWER` | 只读访客 | 仅可查看，不可执行新增、修改、删除、同步、手动检测、告警状态修改、设备授权等写操作 |

第一阶段权限模型采用两层控制：

- 角色控制：后端接口对写操作统一校验 `ADMIN/OPERATOR`
- 数据隔离：设备、视频、告警、检测记录等查询统一按当前用户绑定设备过滤

> 自注册用户默认角色为 `OPERATOR`，用于满足当前 Demo 的自助使用场景。

## 3. 第一阶段范围

### 3.1 已交付能力

- 用户注册、登录、获取个人信息
- 萤石 OAuth 授权与用户设备绑定
- 设备列表、设备详情、设备状态管理
- 实时预览、云录像检索与回放
- 宠物档案管理
- 检测配置与安全区域配置
- 定时检测、检测记录查询、异常行为分析
- 告警同步、告警查询、告警详情
- AI 宠物行为分析与问答

### 3.2 第一阶段不包含

- 企业级租户/组织架构管理
- 消息推送中心（短信、邮件、企业微信、Webhook）
- 完整审计日志
- 批量设备管理和设备分组
- 多宠物追踪与身份识别
- 高可用部署与灰度发布

## 4. 验收口径

第一阶段建议按“Demo 验证通过”而非“生产上线通过”验收，核心标准如下：

### 4.1 功能验收

- 能使用账号登录系统并加载当前用户信息
- 能完成萤石授权并看到当前用户的设备列表
- 能基于设备创建宠物检测配置与安全区域
- 能手动触发检测并生成检测结果
- 能查看检测记录、异常告警和视频流

### 4.2 安全与权限验收

- 未登录不可访问业务接口
- `VIEWER` 不可执行写操作
- 用户 A 无法读取或修改用户 B 绑定设备相关的数据
- 视频、告警、检测记录均按当前用户设备范围隔离

### 4.3 工程验收

- 前端支持 `npm run lint` 和 `npm run build`
- 仓库提供 CI 工作流，覆盖前端 lint/build 与后端 test
- 后端提供最小单元测试入口
- 部署文档、产品说明、接口文档齐备

## 5. 交互说明

### 5.1 设备绑定流程

1. 用户登录系统
2. 进入“绑定萤石设备”
3. 发起 OAuth 授权
4. 开发环境优先回调到前端 `/oauth/ezviz/callback`，再由前端调用后端完成绑定
5. 后端写入 `user_ezviz_account` 与 `user_device`
6. 前端展示当前用户可访问设备

### 5.2 检测配置流程

1. 用户创建宠物档案
2. 选择自己有权限访问的设备
3. 配置安全区域与检测阈值
4. 启用定时检测或手动触发检测
5. 通过记录与告警页查看结果

## 6. 运维与发布说明

### 6.1 本地开发检查

- 前端：`npm run lint`
- 前端：`npm run build`
- 后端：`./mvnw test`

### 6.2 CI 检查

仓库内置 `.github/workflows/ci.yml`：

- `frontend` 任务执行 `npm ci`、`npm run lint`、`npm run build`
- `backend` 任务执行 `./mvnw test`

## 7. 已知限制

- 当前仍是单实例 Demo 架构，未做高并发和横向扩展设计
- 异常分析阈值和告警冷却策略仍偏业务试运行参数
- AI 能力依赖外部平台响应质量，结果仅作为辅助判断
- 设备能力仍以萤石开放平台能力边界为准
